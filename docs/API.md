# API плейграунда: зачем OpenAPI и что с безопасностью

## Что у нас есть

| Артефакт | Где лежит | Кто видит |
| --- | --- | --- |
| `docs/openapi/playground.yaml` | Репозиторий Git | Только те, у кого есть доступ к репо |
| JSON-эндпоинты `/playground/...` | Продакшен / dev | Только **авторизованные** пользователи (cookie + CSRF на POST) |

**Важно:** Django **не** раздаёт папку `docs/` как статику. Файл OpenAPI не открывается по URL вроде `yoursite.com/docs/openapi/playground.yaml`, пока вы сами не настроите отдачу.

Страница плейграунда (`GET /playground/{task_id}/`) — обычный HTML; в OpenAPI её нет намеренно: spec описывает только JSON API для терминала и редактора.

## Зачем вам OpenAPI — реально ли это полезно?

**Да, но не для учеников.** Это внутренняя документация для разработки.

### Когда помогает

1. **Вы и контрибьюторы** — один источник правды: какие поля в `validate`, что возвращает `hint`, формат `task_id` (`gh_1_1`, не `gh.1.1`).
2. **AI-агенты и code review** — меньше расхождений между `playground.js` и бэкендом; проще не сломать контракт при рефакторинге.
3. **Будущие интеграции** — если появится мобильное приложение, CLI или автотесты снаружи браузера, spec уже готов.
4. **Онбординг** — новому человеку не нужно читать весь `playground.py`, чтобы понять surface API.

### Когда не нужен

- Конечным пользователям курса он **не нужен**.
- Если проект один человек и API не меняется годами — можно жить без spec, но тогда контракт живёт только в коде и JS.

**Итог:** полезен как **технический паспорт API для команды**, не как пользовательская документация.

## Нужно ли скрывать от посторонних?

### Что уже защищено

Сами эндпоинты не «публичный REST без авторизации»:

- `@login_required` на всех handler'ах;
- CSRF на POST;
- rate limit на run / files / hint;
- allowlist команд в песочнице (не произвольный shell).

Знание URL `/playground/gh_1_1/validate/` **не даёт** выполнить запрос без валидной сессии залогиненного пользователя.

### Что даёт утечка OpenAPI (если репо публичный на GitHub)

Посторонний увидит **список эндпоинтов и форматы JSON** — не секреты, не ключи, не обход авторизации. Это уровень «посмотрел Network в DevTools на своём аккаунте», только в виде файла.

Скрытие yaml **не усиливает** безопасность API. Усиливает: логин, CSRF, лимиты, политика песочницы (у вас уже есть).

### Практические рекомендации

| Сценарий | Рекомендация |
| --- | --- |
| Приватный репозиторий | Оставить `playground.yaml` в `docs/` — удобно, риска нет |
| Публичный репозиторий | Можно оставить: в spec нет секретов; польза для open-source > минимальный «recon» |
| Параноидальный продакшен | **Не** подключать Swagger UI / Redoc на проде; **не** класть `docs/` в `STATIC_ROOT` / nginx static |
| Секреты | Никогда не класть в OpenAPI: ключи, внутренние admin URL, детали Docker-образа |

### Чего не делать без необходимости

- Не выкладывать интерактивную документацию (`/api/docs/`) на публичный домен — это лишний шум и лишняя поверхность.
- Не считать, что удаление yaml из репо «закроет» API — эндпоинты всё равно видны в JS и DevTools.

## Как поддерживать в актуальном состоянии

При изменении `apps/core/views/playground.py` или URL в `apps/core/urls.py` обновляйте `docs/openapi/playground.yaml` в том же PR.

Проверка глазами: шесть JSON-маршрутов — `run`, `file/read`, `file/write`, `validate`, `reset`, `hint`.
